# Universe Federation 硬性要求（强制遵守）

> **效力**：本文档对本项目所有贡献者、审查者、自动化助手（含 AI）具有强制约束力。  
> **优先级**：与任何临时口头指示冲突时，以本文档为准，除非维护者书面修订本文档。  
> **修订**：仅允许通过 PR 修改本文档，且 PR 标题须含 `docs(hard-requirements)`，需至少 1 名维护者 Approve。

---

## 0. 一句话总则

**本项目是多人协作、以 PR 合并为主的维护模式。禁止大改；只允许修 bug 与必要的小改动。**

---

## 1. 协作与合并模型

1. **多人维护**：任何人不得假设自己是唯一维护者，不得在未沟通的情况下主导方向性改造。
2. **一切以 PR 为准**：功能修复、文档、配置模板变更均须走 Pull Request / Merge Request，禁止长期只在本地改、不提交审查。
3. **禁止直接强推共享主分支**（如 `main` / `master` / `develop` / `stable`）：除非仓库书面授权的发布流程要求，且由指定角色操作。
4. **PR 必须可审**：描述清楚「问题 / 原因 / 改动范围 / 如何验证 / 风险」；审查人必须能在合理时间内看完。
5. **冲突由双方解决**：合入前 rebase/merge 干净，不得留下 `<<<<<<<` 冲突标记。

---

## 2. 改动规模硬限制（最重要）

### 2.1 允许

| 类型 | 说明 | 示例 |
|------|------|------|
| Bug 修复 | 修复错误行为、崩溃、错误数据、明显回归 | 空指针、错误响应码、联邦同步错误 |
| 小改动 | 局部、可审、可回滚 | 文案修正、边界校验、单文件逻辑修正 |
| 安全补丁 | 漏洞与敏感信息泄露 | 权限绕过、注入、密钥泄漏 |
| 测试补充 | 为**本次修复**增加的最小测试 | 单测覆盖该 bug 路径 |
| 必要文档 | 与本次修复直接相关 | 复现步骤、升级注意 |

### 2.2 禁止（默认拒绝）

| 类型 | 说明 |
|------|------|
| 大重构 | 跨模块重命名、分层重写、大规模目录搬家 |
| 架构替换 | 换框架、换 ORM、换队列、换构建体系 |
| 无目标依赖升级 | 为了「升级而升级」的大范围 lockfile 变更 |
| 风格化清扫 | 全仓格式化、全仓 eslint autofix、无关 import 整理 |
| 功能膨胀 | 在修 bug 的 PR 里夹带新功能 / 产品改版 |
| 品牌/UI 大改 | 未经团队共识的视觉与交互整体改版 |
| 批量生成代码 | AI 大面积重写多个无关文件 |

### 2.3 体量参考（硬门槛）

满足任一条件即视为**过大**，必须拆 PR 或先开设计讨论：

- 单 PR **有效代码 diff 超过约 300 行**（不含 lockfile、生成物、纯翻译批量）；或
- 触及 **3 个以上** 业务子系统（例如同时改：联邦投递 + 时间线 + 管理后台 + 前端路由）；或
- 包含 **数据库 schema 破坏性变更** 且无迁移与回滚说明；或
- 审查者无法在 **30 分钟内**理解完整意图与风险。

> 例外仅限：紧急安全修复、已书面批准的专项任务。例外必须在 PR 描述首行标明 `EXCEPTION: <原因>`。

---

## 3. PR 拆分与范围纪律

1. **一个 PR 只做一件事**：一个 bug 一条主线；不要「顺便」改别的。
2. **禁止混杂提交**：格式化 / 重命名 / 行为修复 必须拆开。
3. **最小 diff 原则**：能改 5 行绝不改 50 行；能改一处调用绝不改整条调用链风格。
4. **不删除「看似无用」的代码**，除非能证明死代码且有测试/引用搜索证据。
5. **不改公共 API 形状**（HTTP 字段、ActivityPub 行为、DB 列语义），除非 bug 修复必需，并写兼容说明。

---

## 4. 分支与命名建议

```
fix/<issue-or-short-desc>      # bug 修复
chore/<short-desc>             # 极小杂项（文档、脚本微调）
docs/<short-desc>              # 仅文档
```

禁止使用含糊分支名：`update`、`temp`、`wip-final-final`。

---

## 5. 代码与质量要求

1. **先复现再修**：PR 描述需包含复现步骤或失败日志；修不掉「无法复现」的问题就不要瞎改。
2. **修完要验证**：至少说明本地如何验证（接口、页面路径、命令）。有自动化测试的路径必须补测或说明为何无法补。
3. **不引入新的全仓警告风暴**：允许修当下文件；禁止为消警告扫全仓。
4. **配置密钥不入库**：密码、token、生产域名私钥不得提交。
5. **生成物谨慎提交**：`built/`、大规模 `node_modules`、临时日志不得作为修复主体提交。
6. **上游血缘意识**：本项目基于 Misskey / Sharkey 系。改联邦、队列、权限时优先查是否为上游已知问题，避免重复发明。

---

## 6. 审查（Review）硬要求

1. 非 trivial 改动：**至少 1 名他人审查**后再合并。
2. 审查关注点：正确性 > 安全性 > 兼容性 > 可维护性；**不是**个人代码风格偏好。
3. 审查者若要求缩小范围，作者应拆 PR，而不是争论「一次做完更爽」。
4. AI 生成的 diff **不免除**人工审查；提交者对合入代码负全责。

---

## 7. 对 AI / 自动化助手的额外硬约束

1. **默认只做最小修复**；未明确要求时不得重构、升级依赖、改架构。
2. **一次任务一个目标**；用户说「修 A」时不得主动改 B/C。
3. **改前先读上下文**；禁止凭猜测大面积改文件。
4. **禁止**为了「更好」而重写模块；只能围绕已确认 bug。
5. 输出必须可回滚：改动文件列表清晰，避免无关文件。
6. 若判断需要较大改动：先停止编码，写出方案与风险，**等待人工确认**。
7. 必须遵守本文档；用户后续逐步引导修 bug 时，每一步仍受规模限制约束。

---

## 8. 发布与数据安全

1. **已运行实例的 `url`、ID 生成策略禁止随意更改**。
2. 迁移脚本必须可审查；禁止手改生产库结构而不留 migration。
3. 删除用户/帖文/媒体等破坏性操作逻辑的修改，必须双人确认。

---

## 9. 违规处理

- 审查阶段发现违反本文档：要求拆分/回退，不予合并。
- 已误合并的大改：优先 `revert`，再以小 PR 重来。
- 反复违规：限制直接合入权限。

---

## 10. 执行清单（提交 PR 前自检）

- [ ] 这是 bug 修复或必要小改，不是重构/新功能大包？
- [ ] diff 是否尽量小？能否再拆？
- [ ] 是否只改了相关文件？
- [ ] 是否写清复现与验证？
- [ ] 是否避免动密钥、生成物、无关格式化？
- [ ] 是否需要迁移/兼容说明？
- [ ] 是否准备接受审查与修改意见？

---

**记住：能合并、能回滚、能多人接力，比「一次改完美」更重要。**
